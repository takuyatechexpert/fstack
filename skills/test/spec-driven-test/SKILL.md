# テストコード作成スキル

対象クラスの仕様を正確に理解し、テストケースを設計してからテストコードを実装する。
**実装を通すためのテストではなく、仕様に沿ったテストを書く**ことが最重要原則。

## 大原則

1. **仕様ファーストのテスト**: 実装の内部構造に依存せず、クラスの「振る舞い」と「契約」をテストする
2. **テストケース設計が先**: コードを書く前にテストケース一覧を設計し、ユーザーに提示する
3. **3種のテストは必須**: 正常系・異常系・境界値を必ずカバーする
4. **Arrange-Act-Assert パターン**: 全テストメソッドでAAA パターンを厳守する

## ワークフロー

### Phase 0: プロジェクト構造の把握

テスト作成前に、プロジェクトのテスト環境を把握する。

**実施すること:**
1. テストフレームワークの設定ファイルを探す（`phpunit.xml`, `jest.config.*`, `vitest.config.*`, `pytest.ini` 等）
2. 既存テストディレクトリの構造を確認する
3. 既存テストファイルを1〜2件読み、プロジェクトのテスト規約・パターンを把握する
4. ベースTestCaseクラスがあれば確認する
5. CLAUDE.mdやプロジェクトのドキュメントにテスト方針の記載があれば確認する

**把握すべき情報:**
- テストランナーとバージョン
- テストディレクトリの配置ルール（Unit / Feature / Integration の分類）
- テストファイルの命名規則（`*Test.php`, `*.test.ts`, `test_*.py` 等）
- 共通のセットアップ処理やヘルパー
- モックライブラリ（Mockery, Jest mock, unittest.mock 等）
- テストデータの作成方法（Factory, Fixture, Builder 等）

### Phase 1: 対象クラスの特定と仕様理解

ユーザーからテスト対象のクラスまたはファイルパスを受け取る。

**実施すること:**
1. 対象クラスのソースコードをReadで全文読み込む
2. クラスの責務・公開メソッド・依存関係を把握する
3. 以下を特定する:
   - publicメソッドの一覧と各メソッドのシグネチャ（引数・戻り値の型）
   - 依存クラス（コンストラクタインジェクション、メソッドインジェクション）
   - バリデーションルール（FormRequestなどの場合）
   - 例外がスローされる条件
   - 状態変更（DB操作、副作用）
4. 依存先のインターフェース・クラスも必要に応じてReadで確認する
5. 対象クラスが使用するEnum、ValueObject、DTOなどの関連クラスも確認する

**テスト種別の判定（言語・フレームワーク共通の考え方）:**
- HTTPリクエストを受けるクラス（Controller等） → Feature / Integration テスト
- バリデーション定義クラス（FormRequest等） → Feature / Integration テスト
- CLIコマンドクラス → Feature / Integration テスト
- ビジネスロジッククラス（Service, UseCase, Domain等） → Unit テスト
- データアクセスクラス（Repository, Infrastructure等） → Unit テスト
- 値オブジェクト / DTO → Unit テスト
- ユーティリティ / ヘルパー → Unit テスト

### Phase 2: テストケース設計

対象クラスの**publicメソッドごと**にテストケースを設計する。

**テストケースの3分類（全て必須）:**

#### 正常系（Happy Path）
- メソッドが期待通りに動作するケース
- 代表的な入力値での動作確認
- 複数の正常パターンがある場合は全パターンを網羅
- 戻り値の型・構造の検証

#### 異常系（Error / Exception Path）
- 不正な入力に対する振る舞い
- 例外がスローされるケース（例外クラスとメッセージの検証）
- バリデーションエラー
- 存在しないデータへのアクセス
- 権限エラー
- 依存サービスの障害時の振る舞い

#### 境界値（Boundary）
- 数値の上限・下限（0、1、最大値、最小値）
- 文字列の空文字、最大長
- 配列の空、1件、大量件数
- 日付の境界（期間の開始日・終了日ちょうど、前後1秒）
- null許容フィールドのnull/非null
- bool値のtrue/false両方

**設計のルール:**
- テストメソッド名は、何をテストしているかが明確にわかるようにする（プロジェクトの命名規約に従う）
- 1テストメソッドにつき1つの振る舞いを検証する（複数のassertは可だが、検証対象は1つ）
- テストケース一覧をマークダウンテーブルでユーザーに提示する

**テストケース一覧の出力フォーマット:**

```
## テストケース設計: {クラス名}

### {メソッド名}()

| # | 分類 | テストケース | 期待結果 |
|---|------|-------------|---------|
| 1 | 正常系 | {ケース説明} | {期待する振る舞い} |
| 2 | 正常系 | {ケース説明} | {期待する振る舞い} |
| 3 | 異常系 | {ケース説明} | {期待する振る舞い} |
| 4 | 境界値 | {ケース説明} | {期待する振る舞い} |
```

### Phase 3: テストコード実装

設計したテストケースに基づいてテストコードを実装する。

**テストファイルの配置ルール:**
- Phase 0 で把握したプロジェクトの配置ルールに従う
- 一般的には、ソースコードのディレクトリ構造をテストディレクトリ内でミラーリングする

**実装ルール:**

1. **テストクラスの構造**
   - プロジェクトの既存テストのスタイルに合わせる
   - 共通セットアップ処理はsetUp / beforeEach 等にまとめる
   - テスト対象クラスとの対応関係が明確になるようにする

2. **テストメソッド名**
   - プロジェクトの命名規約に従う（日本語、英語、snake_case等）
   - 何をテストしているかが名前だけで理解できる
   - 例（日本語）: `getにて該当データがある場合に正しく件数を取得できること`
   - 例（英語）: `test_get_returns_correct_count_when_data_exists`

3. **Arrange-Act-Assert**
   - `// Arrange` `// Act` `// Assert`（または同等の）コメントを記載
   - Arrange: テストデータ作成、モック設定
   - Act: テスト対象メソッドの実行（1行が理想）
   - Assert: 結果の検証

4. **モックの方針**
   - 外部依存（DB、API、ファイルシステムなど）はモック化する（Unitテスト時）
   - Integration / Feature テストでは実際のDB等を使用する
   - プロジェクトで使用されているモックライブラリに合わせる
   - モックの期待呼び出し回数も検証する

5. **アサーションの方針**
   - 厳密比較を優先する（`assertSame`, `toBe`, `===` 等）
   - 型が異なる場合のみ緩い比較を使用
   - `assertTrue`/`assertFalse` よりも具体的なアサーションを優先する
   - DB状態の検証にはフレームワーク提供のアサーション（`assertDatabaseHas`等）を活用

6. **データプロバイダ / パラメータ化テストの活用**
   - 同じロジックで入力値だけが異なるテストケースはパラメータ化する
   - 境界値テストで特に有効

7. **テストデータ**
   - プロジェクトのFactory / Fixture / Builder を積極的に活用する
   - テストに必要な最小限のデータのみ作成する
   - マジックナンバーを避け、意図が明確な値を使用する

### Phase 4: 検証

テストコード実装後に以下を確認する:

1. **構文チェック**: 構文エラーがないか確認
   - PHP: `php -l {テストファイルパス}`
   - TypeScript/JavaScript: `npx tsc --noEmit` または ESLint
   - Python: `python -m py_compile {テストファイルパス}`

2. **テストケースの網羅性確認**: Phase 2で設計した全テストケースが実装されているか確認

3. **結果の報告**: 以下のフォーマットで実装結果を報告

```
## テスト実装結果

- テストファイル: `{パス}`
- テストケース数: {N}件（正常系: {n1} / 異常系: {n2} / 境界値: {n3}）
- 構文チェック: OK / NG

### 実装したテストケース一覧
| # | 分類 | メソッド名 | テストケース |
|---|------|-----------|-------------|
| 1 | 正常系 | {テストメソッド名} | {ケース説明} |
| ... | ... | ... | ... |
```

## テスト種別ごとの注意事項

### Controller / HTTP Handler テスト（Feature）
- 認証・認可の設定を行う
- HTTPメソッドでリクエストを送信
- ステータスコード、レスポンス構造、リダイレクト先を検証
- バリデーションエラーのテストも含める

### バリデーション定義テスト（Feature）
- バリデーションルールごとにテストケースを作成
- 正常値・不正値・境界値をパラメータ化テストで網羅
- 実際にリクエストを送信して検証

### ビジネスロジックテスト（Unit）
- 依存をモック化
- 入出力を厳密に検証
- エッジケース（空配列、null、0件）を重点的にテスト
- 処理フロー（呼び出し順序・引数）の検証

### ValueObject / DTO テスト（Unit）
- 不変性の検証
- 等価性の検証
- 生成時の制約（バリデーション）の検証
- 変換メソッドの検証

### CLI Command テスト（Feature）
- コマンドを実行
- 終了コード、出力メッセージを検証
- DB状態やファイル出力の変更を検証
