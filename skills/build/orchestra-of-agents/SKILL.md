# orchestra-of-agents

agent teams を活用した並列実装・テスト・レビューワークフロー。
Wave 方式で依存関係を考慮しながら並列実装し、仕様ベーステストと構成に応じたレビュー体制で品質を担保する。

## 参照ファイル

実行時に必ず Read で読み込むこと。要約やメモリからの再現は禁止。

- **エージェントプロンプト集**: `${CLAUDE_SKILL_DIR}/references/agent-prompts.md`
- **標準構成ワークフロー詳細**: `${CLAUDE_SKILL_DIR}/references/standard-workflow.md` — 標準構成の場合のみ参照
- **最小構成ワークフロー詳細**: `${CLAUDE_SKILL_DIR}/references/minimal-workflow.md` — 最小構成の場合のみ参照

## 構成の選択

タスク数に応じて **標準構成** または **最小構成** を選択する。

| 条件 | 構成 | エージェント数 | レビュー体制 |
|------|------|---------------|-------------|
| 6タスク以上 | 標準構成 | 4体 + leader | クロスレビュー + 横断レビュー |
| 5タスク以下 | 最小構成 | 5体 + leader | ダイレクトレビュー + 総合判断 |

実装規模が小さくても、レビューとテストの厚みは維持する。

---

## 標準構成（6タスク以上）

### チーム構成

| エージェント | 役割 |
|---|---|
| leader | オーケストレーター。タスク分析・分割・Wave 管理・メッセージ中継を行う |
| impl-1 | 基盤実装 → ページ実装 → クロスレビュー |
| impl-2 | API実装 → ページ実装 → クロスレビュー |
| tester | 仕様ベーステスト作成（/spec-driven-test） |
| reviewer-c | 横断レビュー + テスト品質確認 |

leader はオーケストレーター自身が兼任する（別エージェントとして spawn しない）。

### impl 数の調整

Wave 方式により基盤と応用を分離するため、impl は原則 **2名固定**。

- **拡大**: Wave 2 のタスクが **15ファイル超/impl** になる場合、impl-3 を追加可能。ただし impl-3 は **Wave 2 からの参加**（Wave 1 の成果物を前提とするため、Wave 1 には参加しない）

### ワークフロー

詳細手順は `${CLAUDE_SKILL_DIR}/references/standard-workflow.md` を参照。

```
Wave 0: leader が設計契約書を作成                           ← 設計合意
Wave 1: impl-1(基盤) + impl-2(API) + tester(テスト作成)    ← 3並列
Wave 2: impl-1(応用A) + impl-2(応用B) + テスト実行          ← 2並列+検証
Wave 2.5: leader がビルド＋起動スモークテスト               ← 統合検証
Wave 3: クロスレビュー + reviewer-c(横断レビュー)            ← 3並列
Wave 4: 最終判断 + 修正ループ（最大3回）
```

## 最小構成（5タスク以下）

### チーム構成

leader（兼任）+ impl-1, tester, reviewer-a（実装レビュー）, reviewer-b（テストレビュー+実行）, reviewer-leader（総合判断）の **計5体 spawn**。reviewer-leader は Wave 3 で初めて spawn（同時最大4体）。各プロンプトは `agent-prompts.md` 参照。

### ワークフロー

詳細手順は `${CLAUDE_SKILL_DIR}/references/minimal-workflow.md` を参照。

```
Wave 0: leader が設計契約書を作成                                ← 設計合意
Wave 1: impl-1(実装) + tester(テスト作成)                        ← 2並列
Wave 2: reviewer-a(実装レビュー) + reviewer-b(テストレビュー+実行) ← 2並列・直接引き渡し
Wave 2.5: leader がビルド＋起動スモークテスト                     ← 統合検証
Wave 3: reviewer-leader(総合判断) → 直接差し戻し or OK            ← 節目報告
```

レビューフェーズ（Wave 2-3）では**直接引き渡し**方式を採用。reviewer ↔ impl の直接質問を許可し、leader には節目報告で状況を共有する。

## Wave 0: 設計契約（全構成共通）

Wave 1 開始前に leader が**インターフェース契約書**を作成する。並行実装のモジュール境界不整合を防ぐため、型レベルだけでなくセマンティクスレベルの合意を取る。

### 契約書に含める内容

1. **エンティティ関係図** — 主要な構造体/型の関係と ID の流れ
2. **モジュール境界の API** — impl 間の呼び出しインターフェース（関数シグネチャ + 戻り値の意味）
3. **データフロー** — イベントがどの順序でどのモジュールを通るか
4. **共有型の定義場所** — どの型をどのモジュールに置くか、重複定義の禁止ルール

### 手順

1. leader がタスクを分析し、上記4項目の契約書を作成
2. 契約書を `/tmp/orchestra-{team-name}/design-contract.md` に保存
3. **architect エージェント**（`architect.md`）を spawn し、契約書の評価を依頼（ID紐付け漏れ・データフロー不整合・型重複・API整合性の観点）
4. architect の指摘を反映して契約書を改善（最大2回のレビューループ）
5. impl への spawn 時に契約書のパスを渡し、参照を義務付ける

---

## Wave 2.5: 統合スモークテスト（全構成共通）

Wave 2 完了後、Wave 3 開始前に leader がビルド＋テスト実行を行う。**問題発見時は必ず impl に差し戻す（自分で修正しない）。**

1. ビルド確認 → 起動テスト（可能なら）→ 不可ならスキップ
2. タイムアウト: ビルド5分、起動30秒。リトライ最大2回

---

## 共通事項

### leader の役割

タスク分析・分割・Wave 管理・スモークテスト・最終報告を担当する。

**Edit / Write の使用は禁止。** 問題発見時は必ずメンバーに差し戻す。「ちょっとした修正」も禁止（v2.0/v3.0 で leader が作業を巻き取る問題が繰り返し発生したため）。

### tester への指示方針

「何をテストするか」は仕様から、「どうアサートするか」は既存テストのパターンから学ぶ。tester への指示には **既存テストファイルのパス** と **外部データのサンプル** を含めること（v3.0 P1: 仕様からの推測で JSON 構造を誤りテスト偽失敗が発生）。

### クリーンアップ

1. 全メンバーに `shutdown_request` → `TeamDelete` → 最終レポート表示
2. 実行ログを `/tmp/orchestra-{team-name}/{YYYY-MM-DD}-{task-slug}.md` に保存（テンプレート: `${CLAUDE_SKILL_DIR}/references/run-log-template.md`）
3. シャットダウン無応答時: 再送1回 → それでも無応答ならユーザーに手動対応依頼

### 修正ループ時のテスト修正責務

レビュー修正で実装が変わった場合、テスト修正は **tester が行う**（impl は禁止）。テストは仕様の番人であり、tester が仕様と照合して更新することで修正の正しさを検証するゲートになる。impl に任せると「実装に合わせてテストを歪める」リスクがある（v4.0）。

### 注意事項

- **impl の担当ファイルは完全に分離する** — 同じファイルを複数の impl が編集すると競合が発生する。Wave 1 のタスク分割時に leader がファイル単位で明確に割り当てること
- **tester は仕様を入力としてテストを書く** — アサーション方法は既存テストのパターンを参考にする
- **修正ループ時、impl はテストファイルを編集しない** — tester の責務（上記参照）
- **Wave 間の移行は全員完了を確認してから**
- **レビューエージェントはレビュー Wave で初めて spawn する**
- **修正ループ3回超えたら人間にエスカレーション**
- **タスク分割はファイル単位が原則** — 同一ファイルの並行編集は禁止。密結合の場合のみ機能単位を許容
- **最低3タスク同時ルールは標準構成のみ**
