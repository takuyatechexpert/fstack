# Opus 4.7 プロンプト/スキル作成ガイド

> Opus 4.6 から 4.7 で変わった挙動と、4.7 で新規にプロンプト・スキル・ルールを書くときの実務ガイド。

Opus 4.7 は 4.6 と比べて推論特性・指示追従の傾向・ツール提供形態が変わっている。4.6 向けに最適化した書き方を 4.7 にそのまま流用すると、性能が出にくいケース・新機能を活かせないケース・失敗ループに入るケースがある。

このドキュメントは、

- **既存資産（4.6 時代に書いたスキル）を 4.7 で動かすときに注意する点**
- **4.7 で新規にスキル・プロンプトを書くときの推奨パターン**

を整理する。

---

## 1. 4.6 → 4.7 で変わった主要ポイント

### 1.1 ツール提供形態: deferred tools の登場

4.7 では **一部のツールが deferred（schema 未読込）状態でセッション開始する**。例: `TaskCreate` `WebFetch` `WebSearch` `LSP` `EnterPlanMode` 等。

- 初回利用前に `ToolSearch(query: "select:<name>")` で schema を取得する必要がある
- 4.6 ではすべてのツール schema が最初から読み込まれていたので、この手順は不要だった
- スキル本文で `TaskCreate` 等の deferred tool を呼ぶ前提なら、利用前に ToolSearch を呼ぶ手順を本文に明記する

> 詳細: `rules/context-management.md` の「Deferred Tools と ToolSearch」

### 1.2 Auto Memory（構造化メモリ）が標準機能化

4.7 はセッションを跨ぐ構造化メモリを **標準機能として保持する**。`~/.claude/projects/<cwd-hash>/memory/` 配下に user / feedback / project / reference の 4 種類で保存される。

スキル設計への影響:

- ユーザーの好み・ガイダンスを SKILL.md に静的に書き込まなくてよい（自動記憶される）
- 「以前のセッションで〜と決めた」系の文脈は memory が拾うので、スキル冒頭で改めて伝える必要は減った
- 一方で **memory に書かれた内容と SKILL.md が矛盾すると揺らぐ**。スキル側は普遍ルールに絞り、ユーザー固有設定は memory に任せる

> 詳細: `rules/memory.md`

### 1.3 Plan Mode の標準化

`EnterPlanMode` / `ExitPlanMode` が標準提供される。実装フェーズの前に **編集を抑制した計画立案フェーズに入る装置**。

- 4.6 では「計画してから実装」を文章で促す必要があったが、4.7 ではモードとして分離できる
- 5 ファイル以上を編集する大規模スキル・破壊的変更を扱うスキルは Plan Mode 起動を本文に組み込む

> 詳細: `rules/plan-mode.md`

### 1.4 指示追従がより文字通りになった

4.6 と比べて、4.7 は **指示を字義通りに解釈する傾向が強い**。

- 否定形（「〜しない」「〜禁止」）は禁止リストとして強く拘束する
- 一方で「では何をすべきか」が抜けていると行動が止まりやすい
- 曖昧表現（「適切に」「必要に応じて」）の解像度が下がる

→ 否定形は「絶対ガード」のみに残し、それ以外は **肯定形＋受け入れ基準** で書き直すと挙動が安定する。

### 1.5 並列ツール呼び出しがデフォルト挙動として強化

独立した tool call は単一メッセージ内で並列発行されやすくなった。逆に「順番に読んで」のような逐次指示は冗長になる。

- 「複数の Read を並列で」と書かなくても並列発行されるケースが増えた
- ただし **依存関係のある呼び出しまで並列化されると壊れる** ので、明示的に「A の結果を見てから B」と書く

### 1.6 コンテキスト管理: 1M 窓と圧縮

4.7 のうち 1M context window モデルは長大なやり取りに耐えるが、圧縮が走るタイミング・キャッシュ TTL は依然として実務上の制約。

- プロンプトキャッシュは 5 分 TTL — 270 秒以内のループ vs 1200 秒以上の待機を意識する
- `/handover` は 65% 付近で発火させる方針を維持

## 2. 4.7 でスキル・プロンプトを書くときの推奨パターン

新規作成時、または 4.6 時代のスキルを 4.7 用に整える際のチェックリスト。

### 2.1 否定形は「絶対ガード」のみ、それ以外は肯定形

絶対ガードとして残すもの（曖昧化させない）:

- 本番ブランチへの push 禁止
- 金銭処理・破壊的操作の制限
- セキュリティ関連（シークレットの平文出力禁止 等）

それ以外は肯定形に書き換える:

```diff
- テストファイルは編集しない
+ テストファイルは Read のみ。変更が必要な場合は user に確認後に Edit する
```

### 2.2 「タスク委任の 5 点セット」を最初のターンで揃える

4.7 は最初のターンで完結した文脈を渡されると質量ともに伸びる。1 行ずつ指示するペアプロ的構成を避ける。

- **意図** — 何を達成したいか
- **制約** — 守るべき規約・触ってはいけない範囲
- **受け入れ基準** — 完了判定条件
- **関連ファイル** — 既存コードの場所（`path:line` 形式）
- **検証手段** — 実行すべきテスト/lint コマンド

### 2.3 自己検証手段をコマンドレベルで明記

```diff
- 実装後にテストが通ることを確認する
+ 実装後に以下を実行:
+   1. `make test` (exit 0 を確認)
+   2. `make lint` (exit 0 を確認)
+   3. LSP diagnostics でエラーゼロを確認
```

判定基準が曖昧だと「通ったつもり」で完了宣言する挙動は 4.7 でも変わらない。

### 2.4 deferred tool を使うスキルは ToolSearch 手順を組み込む

スキル本文の前段に以下のような注記を入れる:

```markdown
## 前提

このスキルは `TaskCreate` `LSP` を使用します。
セッション内で初回利用前に以下を実行してください:

  ToolSearch(query: "select:TaskCreate,LSP")
```

### 2.5 並列化は「依存関係がない」ことを明示

4.7 はデフォルトで並列発行しやすいが、**依存関係を明示しないと壊れる**ケースがある:

```diff
- ファイル A, B, C を読み込み、A の内容に応じて B を解析する
+ まずファイル A を Read。A の内容を確認した後、B と C を **並列で** Read する
```

### 2.6 SKILL.md は 200 行以内、詳細は references/ へ

長大な SKILL.md は context を圧迫する。本文には **判断基準とワークフロー** を残し、テンプレート・チェックリスト・詳細手順は `references/*.md` へ外出しする。

```markdown
詳細な検証チェックリストは `${CLAUDE_SKILL_DIR}/references/verification-checklist.md` を参照する
```

ただし **ワークフローの中核を外出しすると毎回 Read が走って遅くなる**。本文に残すもの・外出しするものの判断は次の通り:

| 残す | 外出しする |
|---|---|
| 起動条件・適用範囲 | 詳細な手順テンプレ |
| 主要な判断基準 | エラー対応リスト |
| 自己検証コマンド | 過去事例・サンプル |

### 2.7 frontmatter の description は慎重に扱う

`description` はスキル選択ロジックに直結する。本文を書き換える際に description を同時に触ると、スキルが起動しなくなる/別シーンで誤起動するリスクがある。

- **本文と description は別フェーズで編集する**
- description 変更後は代表シナリオで起動条件を再検証する

### 2.8 model 指定は機能要件由来

`model: haiku` 等の指定は速度・コスト・機能要件由来。**4.7 へのアップデート理由だけで変更しない**。

### 2.9 CLAUDE.md ルールとの重複を避ける

グローバルルール（テスト方針・git ワークフロー・破壊的操作禁止 等）と同じ内容をスキル内に重複記載しない。スキル側は **「該当ルールを参照する」と書くだけ** に絞る。

```markdown
コミット規約は `~/.claude/rules/git-workflow.md` に従う
```

## 3. 4.6 時代のスキルを 4.7 で動かすときに陥りがちな症状

| 症状 | 主な原因 | 対処 |
|---|---|---|
| スキル冒頭で「TaskCreate が見つからない」エラー | deferred tool の schema 未取得 | 本文に ToolSearch 手順を追記 |
| 同じエラーで失敗ループ | 否定形指示で「次に何をすべきか」が抜けている | 否定形 → 肯定形＋受け入れ基準 |
| 完了宣言が早い・取りこぼしが多い | 検証手段が曖昧 | コマンドレベルで明記 |
| ユーザーの好みが毎回反映されない | スキルに静的に書いてしまっている | memory に委ね、スキルは普遍ルールに絞る |
| 5 ファイル以上の編集で途中停止 | Plan Mode を使っていない | EnterPlanMode 起動を本文に組み込む |
| 過剰な並列発行で API が壊れる | 依存関係を明示していない | 「A の後に B/C を並列」と段階分け |

## 4. 共通の落とし穴

- **description 変更による起動条件のずれ** — 本文と description は別フェーズで扱う
- **絶対ガードの肯定形化** — 「本番に push しない」を「develop に push する」と書き換えると別ブランチへの push を許容する余地が生まれる。ガード系は否定形のまま残す
- **references/ への過度な外出し** — ワークフロー中核は本文に残す
- **CLAUDE.md と重複** — スキル側は参照のみに絞る
- **ハードリンク/シンボリックリンクの見落とし** — fstack のように setup でリンクを張っている場合、書き換え前に `ls -la` / `readlink` で実体パスを確認する
- **memory と SKILL.md の矛盾** — 4.7 では memory が静かに挙動を左右する。スキルが意図通り動かないときは `MEMORY.md` を確認する
