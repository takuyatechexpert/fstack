# Plan Mode 運用

## 概要

Claude Opus 4.7 で `EnterPlanMode` / `ExitPlanMode` が標準提供される。
Plan Mode は **編集・コマンド実行を抑制し、計画立案に集中する思考フェーズの仕組み**。
読み取り・調査ツールのみ使える状態にすることで、実装に踏み込む前の意図合わせを徹底する。

## レイヤーの整理（並列実装系スキルとの違い）

Plan Mode と並列実装系スキル（`/orchestra-of-agents` 等）は **異なるレイヤー** で、両立する:

| 局面 | 役割 | ツールの使い方 |
|---|---|---|
| Plan Mode | 計画立案フェーズ — 何をどう実装するかの意図合わせ | 読み取り・調査のみ。Edit/Write は使えない |
| 並列実装系スキル | 実装フェーズ — 計画に沿って Wave/Team で並列実行 | エージェントを駆動して並列編集・テスト |

> 大規模変更は **Plan Mode で計画 → ユーザー承認 → 並列実装系スキルで実行** という流れが理想。
> Plan Mode は実装手段を縛るものではなく、計画と実装の境界を作る装置。

## いつ Plan Mode を使うか

### 使うべき

- 5ファイル以上の編集を伴う変更
- 既存アーキテクチャに影響する変更（DI 設計・モジュール境界の移動）
- 破壊的変更（API 互換性・DB スキーマ）
- 複数の選択肢があり、トレードオフを整理する必要がある変更

### 使わなくてよい

- 単一ファイル内の編集（typo・1関数の修正）
- 明確に1択の修正（バグ FIX で原因が確定済み）
- 探索・調査タスク（`Explore` サブエージェントへ）

## ワークフロー

1. **EnterPlanMode** で計画モードに入る
2. 既存コード読み取り・依存調査・候補比較
3. ユーザーへ計画提示（複数案＋推奨案）
4. 承認後 **ExitPlanMode** で実装フェーズへ移行

## 計画の品質基準

提示する計画には最低以下を含める:

- **意図** — 何を達成するか
- **影響範囲** — 触るファイル・モジュール・テスト
- **手順** — Step に分解（並列化が見えてきたら次フェーズで Wave 設計に渡す）
- **リスク** — 失敗パターン・ロールバック手段
- **検証** — `testing.md` の検証手段から選択

## エージェントとの関係

| 対象 | Plan Mode 連携 |
|---|---|
| `architect` agent | Plan Mode 内で読み取り専用解析を委譲する用途。計画案を作る相棒 |

## 計画変更時のルール

実装中に方針変更があった場合:

- 軽微な調整 → そのまま続行、最後に変更点を報告
- 大幅な方針変更 → 一度 ExitPlanMode して再度 EnterPlanMode し直す
