# Auto Memory 運用

## 概要

Claude Opus 4.7 以降の **標準機能** として、セッションを跨いで記憶される構造化メモリが提供される。
Claude が自律的に書き込み、次回以降のセッションで自動参照する。
保存場所は `~/.claude/projects/<cwd-hash>/memory/`、インデックスは `MEMORY.md`（200行制限、1エントリ1行）。

利用者側で特別な設定や雛形作成は不要。**機能が無効化されている場合のみ、有効化を推奨する**。

## 4種類のメモリタイプ

### user
- ユーザーの役割・経験・好み・知識領域

### feedback
- 「こうして」「やめて」のガイダンス（**Why** と **How to apply** 必須）
- 構造: ルール → **Why:** 理由 → **How to apply:** 適用条件

### project
- 進行中の作業・締切・関係者（時間で陳腐化）
- 相対日付は絶対日付に変換（「来週木曜」→「YYYY-MM-DD」）

### reference
- 外部システム・リソースへのポインタ

## 保存しないもの

- コードパターン・規約・アーキテクチャ・ファイルパス（コードを読めば分かる）
- git 履歴・最近の変更（git log で十分）
- デバッグ解法・修正レシピ（commit message に context あり）
- CLAUDE.md / rules に既出の内容
- 進行中タスクの一時状態（タスク管理は TaskCreate）

## ステール検出ルール

メモリが特定の関数名・ファイルパス・フラグを含む場合、それは **書かれた時点での主張**。
ユーザーの行動に影響する recommendation を出す前に確認する:

1. ファイルパス → 存在確認
2. 関数名・フラグ → grep / LSP `workspaceSymbol` で確認
3. 古ければメモリ更新 or 削除

## crystallize との境界

| 観点 | Auto Memory（純正機能） | knowledge ベース（`/crystallize`） |
|---|---|---|
| 寿命 | セッションを跨ぐが揮発しやすい | 長期資産 |
| 粒度 | 個別の判断・好み | 体系化された知見 |
| 形式 | 1事象1ファイル | 知識領域別の統合ドキュメント |
| 更新 | 日次〜週次で陳腐化 | 月次〜四半期で見直し |

`/crystallize` 実行時:
1. memory のうち繰り返し現れたパターン・体系化可能なものを抽出
2. knowledge へ昇華
3. 蒸留済み memory はクリーンアップ

## ストレージとプライバシー

- 保存場所: `~/.claude/projects/<cwd-hash>/memory/`（Claude Code 純正機能が管理）
- ローカル専用、git 管理外
- プロジェクト固有メモリは `cwd-hash` 単位なので、リポジトリごとに別管理になる

## Auto Memory が動作していない場合

セッションを跨いだ記憶が一切残らない・参照されない様子であれば、Claude Code 側で機能が無効化されている可能性がある。
利用者に有効化を促し、有効化後に `/crystallize` の活用を案内する。
