# テスト

## TDD サイクル

新規機能の実装時は以下のサイクルに従う：

1. **RED** — 失敗するテストを先に書く
2. **GREEN** — テストを通過する最小限の実装をする
3. **REFACTOR** — テストを維持しながらコードを改善する

各ステップで必ずテストを実行し、期待通りの結果を確認してから次に進む。

## カバレッジ目標

- ユニットテスト: **80% 以上**
- 新規機能: 必ずテストを先に書く
- バグ修正: 再現テストを書いてから修正する

## 検証手段（セルフ・ベリフィケーション）

実装・編集後は対応する検証コマンドを実行し、結果を確認してから完了とする。
タスク委任時はこの表から該当コマンドを受け入れ基準として明示する。

| 言語・種別 | 典型的な検証コマンド |
|---|---|
| PHP | `composer test` / `vendor/bin/phpunit` → `vendor/bin/phpstan` or `vendor/bin/psalm` |
| TypeScript / Node | `pnpm test` → `pnpm typecheck` → `pnpm lint`（npm/yarn 環境ではそれぞれ置換） |
| Python | `pytest` → `ruff check` → `mypy`（プロジェクトの設定に従う） |
| Go | `go test ./...` → `go vet ./...` |
| Rust | `cargo test` → `cargo clippy` |
| 共通 | 編集直後に LSP 診断を確認しエラーを解消 |

利用するプロジェクトに応じて、表に行を追加して拡張してよい（例: Ruby `bundle exec rspec` → `rubocop`、Java `./gradlew test` → `./gradlew check` 等）。

## テストケース設計

以下の観点でテストケースを網羅する：

- **正常系** — 期待される入力と出力
- **異常系** — 不正な入力・例外・エラー条件
- **境界値** — 最小値・最大値・空・null・ゼロ
- **パターン網羅** — データプロバイダやパラメタライズドテストを活用し、入力パターンを網羅的にテストする

## モック方針

- **外部 API** — テストダブル（モック・スタブ）でレスポンスを固定する。未登録のリクエストが発生しない仕組みを導入する
- **インターフェース** — モックオブジェクトで振る舞いを定義し、DI コンテナに登録する
- **データベース** — テスト用の実 DB インスタンスを使用する（モック禁止、スキーマ齟齬の防止）。各テスト後にロールバックまたはクリーンアップする
- **キャッシュ・ストレージ** — テスト前後でクリーンアップし、テスト間の状態汚染を防ぐ
- **ファイルシステム** — テスト用一時ディレクトリを使用する
- **時刻依存** — テスト用の時刻固定化機構を使用する（言語・フレームワークが提供するテストヘルパーを活用）

## テストデータのセットアップ

プロジェクトに合わせて以下のパターンを使い分ける：

- **Factory / Builder パターン** — テストデータの生成を専用クラスに委譲する。バリエーションはメソッドチェーンやステートで表現する
- **Fixture / Seed データ** — 静的なテストデータを事前に用意する
- **インラインセットアップ** — シンプルなケースではテストメソッド内で完結させる

## テストの品質

- テストの説明は「何を・どうしたら・どうなるか」を明確に記述する
- 1テスト1アサーション を基本とする（複数アサーションは関連する検証のみ）
- 厳密比較を優先する（型と値の両方を検証）
- テスト間の依存・実行順序依存を作らない
- テストデータのセットアップは各テスト内で完結させる（共有状態を避ける）

## 既存スキルとの連携

- 仕様ベースのテスト作成は `/spec-driven-test` スキルを使用する
- テストファーストの実装は `/orchestra-of-agents` 内でも TDD サイクルを適用する
